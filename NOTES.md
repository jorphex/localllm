At `-b 128 -ub 64`, idle VRAM was about `5318 MiB used / 4690 MiB free`, after short probes it stayed about `5390 / 4618`, and a heavier `~32.5k`-token prompt ingest completed cleanly at about `126.9` prompt tok/s with VRAM still flat around `5396 / 4612`. Impact: this is the first Nemotron 30B profile on this host that looks genuinely usable at `64k` without razor-thin OOM risk.
[Nemotron MoE tuning] outcome — partial MoE CPU offload matters, but only at the high end on this card: plain `64k` hybrid fit had only about `1.1 GiB` free, `--n-cpu-moe 26` still OOMed at load, `--n-cpu-moe 40` still left only about `816 MiB` free after probes, and `--cpu-moe` bought huge headroom (`~6.8 GiB` free) but gave up too much GPU residency. Impact: for this 30B A3B checkpoint on a 10 GB card, `--n-cpu-moe 48` is the practical balance point; smaller partial values are not enough and full `--cpu-moe` is the safety fallback, not the preferred performance profile.

# Lessons
[Residency clarification rule] outcome — when the user asks to stage auxiliary models, confirm CPU-vs-GPU residency and whether the main baseline should stay loaded before treating “load both” as a GPU co-residency request. Impact: avoiding that assumption prevents wasted tuning work and contradictory notes.
[Replay conversion rule] outcome — raw `opencode export` assistant messages are not replay-safe against `llama.cpp` as-is. The replay converter must normalize finish reasons, drop assistant text that accompanies tool calls from history, and add a continuation user prompt whenever transcript-derived history would otherwise end on a tool result or assistant turn. Impact: transcript-derived replay fixtures should be generated through `export_to_fixture.py`, not assembled manually from raw export JSON.
[Evaluation rule] outcome — do not use `max_tokens` caps by default for viability comparisons or real agent tests. Impact: capped runs are valid only when the goal is truncation behavior; otherwise they distort model conclusions.
[Scratch benchmark isolation] outcome — stop the managed main service before scratch GPU tests and keep only one GPU-backed chat model loaded at a time. Impact: this avoids contaminated fit, preload, and VRAM results.
[Compare harness rule] outcome — the OpenCode compare runner must support candidate subsetting and run uncapped requests with the same launch profile for both models. Impact: dry-run sanity checks can stay short without forking the harness logic or reintroducing `max_tokens`.
[Simulation suite rule] outcome — sandbox comparisons should copy a fixture repo into a temp workspace, restrict tool paths to that workspace, and normalize safe test-command variants like `python3 -m unittest ...` and `pytest ...`. Impact: repeatable agent comparisons can be run safely even when the model improvises slightly different test command spelling.
[Simulation scoring rule] outcome — harder sandbox tasks should score both task success and patch scope, because “passes tests” alone hides cases where a model explores or edits the wrong file before converging. Impact: aggregate comparison summaries should keep `expected_files_only` or an equivalent scope-discipline signal, not just pass/fail and latency.
[Qwen request shape] outcome — Qwen 3.5 family requests on this host should set `chat_template_kwargs.enable_thinking` explicitly, keep `tool_choice:"auto"` unless a tool is already chosen, and preserve tool-call history exactly across turns. Impact: request-shape mistakes are a common source of false model-quality regressions.
[Text-only preset rule] outcome — text-only presets must set `MAIN_MMPROJ=` explicitly unless `serve-main.sh` is hardened to stop default-projector inheritance. Impact: omitting that variable can accidentally reuse a multimodal projector from a previous preset.
[GPT-OSS revisit rule] outcome — if GPT-OSS is revisited, use the copied GPT-OSS harnesses and Harmony-style request shaping with a `developer` message and `chat_template_kwargs.reasoning_effort`, not the generic Qwen-style harness. Impact: GPT-OSS comparisons should not be rerun through the generic path by mistake.
[MoE fit rule] outcome — on recent `llama.cpp` builds, `--n-cpu-moe N` and `--cpu-moe` are valid first-line tuning levers for MoE checkpoints when VRAM headroom is the real constraint, but medium partial values can still OOM or leave too little safety margin. Impact: for MoE tuning on this host, test high `N` values first and treat `--fit-target` as secondary because it did not materially change the Nemotron 30B `64k` fit.
[Benchmark isolation rule] outcome — the generic benchmark wrappers only unload the managed main service; they do not automatically kill ad-hoc scratch servers already occupying `8091`. Impact: before any scratch GPU benchmark pass, explicitly stop both the managed main service and any manually launched `llama-server` processes on `8091`, or the benchmark candidate may fail load and produce false results.
[Replay fixture rule] outcome — replay fixtures should encode the actual expected turn shape for the chosen baseline or transcript source, not an aspirational tool sequence from a different harness. Impact: otherwise the replay runner will correctly flag mismatches that come from fixture drift instead of model regressions.
[OpenCode timeout rule] outcome — the current `benchmarks/opencode_compare/run_compare.sh` wrapper does not recover cleanly when a candidate request hits the `COMPARE_TIMEOUT` curl limit; the timed-out run left the managed main service unloaded until it was restored manually and the shell trap also referenced an unset `pid`. Impact: uncapped OpenCode comparisons on long-reasoning candidates should be treated as partial evidence unless the wrapper is hardened first, and future runs should either cap the request budgets or fix the timeout/restore path before using the suite for decisive comparisons.
[HF filename collision rule] outcome — some GGUF finetune repos publish generic filenames like `Qwen3.5-9B.Q4_K_M.gguf`, which will overwrite each other if downloaded straight into the shared `models/` directory. Impact: when staging multiple same-family finetunes, download to separate temp directories first and then copy into `models/` under explicit local names before benchmarking.
