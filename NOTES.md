# Active
[OpenWendy boundary] outcome — `localllm` kept the real legacy fallback behavior for old OpenWendy binary/model roots in `llama_runtime.py` and `scripts/common.sh`, kept the tests that prove those fallbacks, renamed the code/test language to `legacy` rather than `openwendy`, and trimmed the README to one compatibility section plus one brief consumer note. Focused `ruff` and `unittest` verification passed, so the repo now reflects the shared-runtime boundary more cleanly without lying about the compatibility behavior it still owns.
[Tuning review] managed runtime now accepts global and per-service extra `llama-server` args — enables the same tuning flags as shell/systemd launches and preserves service-specific override order.
[Benchmark result] main service runs better with `--threads-http 6 -ctk q8_0 -ctv q8_0` added to the existing profile — reduced simple chat probe from ~0.57 s to ~0.13 s and increased free VRAM from ~262 MiB to ~594 MiB without breaking health checks.
[Slot scheduling] `-np 1` means one `llama-server` slot, and overlapping requests are deferred in the server task queue until the slot is free — `localllm` adds no extra rejection layer, so concurrency is serialized unless the client disconnects or an explicit no-slot probe is used.
[GPU upgrade guidance] outcome — for this `llama.cpp`-style local setup, the practical upgrade from a 10 GB RTX 3080 is primarily more VRAM with a stable CUDA path; 24 GB RTX 3090 is the strongest default value if priced near AMD Radeon AI PRO R9700, while 32 GB R9700 only wins when the operator accepts Linux-first ROCm/Vulkan tradeoffs to maximize model fit.
[3090 vs R9700 tradeoff] outcome — official specs make the decision concrete: RTX 3090 offers 24 GB GDDR6X, 350 W board power, and ~936 GB/s peak bandwidth, while Radeon AI PRO R9700 offers 32 GB GDDR6, 300 W TBP, 640 GB/s bandwidth, Linux-first ROCm support on specific distros, and Linux-only ECC memory support; choose 3090 for the least-risk LLM path and R9700 only when the extra 8 GB directly changes target model sizes enough to justify the software tradeoff.
[Reddit and model-fit scan] outcome — community reports still treat 3090 as the predictable single-card sweet spot for local LLMs, while R9700 feedback is mixed between strong llama.cpp/Vulkan or ROCm results and substantial setup, noise, or throttling complaints; current Hugging Face GGUF sizes show 24 GB already covers strong 24B and many 30B/32B Q4-Q5 models, while 32 GB mainly unlocks higher-bit 32B or extra context headroom rather than an entirely new carefree model tier.
[GPU fault] outcome — the current outage is not a project/runtime bug but an NVIDIA driver/hardware event: kernel logged `Xid 79` at `2026-03-15 06:59:44 UTC` with `GPU has fallen off the bus`, `nvidia-smi` reports unknown device-handle errors, and the stuck `llama-server` processes were terminated to leave the box ready for reboot-level recovery.
[GPU recovery] outcome — after reboot the RTX 3080 returned cleanly: this boot has no fresh `Xid` logs, `nvidia-smi` reports `P8`, about `13 W`, `35 C`, `0%` fan, and no active `llama-server` processes, which is a normal idle recovery state rather than evidence of an immediately persistent fault.
[Systemd autostart] outcome — the model services were installed, enabled, and started on boot, but both crashed in a restart loop because `llama-server` under systemd could not locate sibling `libmtmd.so.0`; exporting the binary directory into `LD_LIBRARY_PATH` from the shared launcher fixed the root cause, and both `localllm-main.service` and `localllm-embedding.service` now stay up and return `/health = 200 OK`.
